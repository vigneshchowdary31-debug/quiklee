🗄️ Database Architecture

Relational Design & Single Source of Truth

The Quiklee database serves as the foundation for the inventory status updates, historical stock logs, and low-stock alerts.

Relational Storage Engine

  • Development: Local SQLite adapter interface.
  • Production Scale: Fully scalable **PostgreSQL** hosted securely on Railway.
  • Protection: Prepared parameterized query layers to block SQL injection vulnerabilities.

Pooling & Adapters

  • Connection Pool: Configured client queues to handle concurrent read/write transactions smoothly.
  • MySQL Adaptability: Custom adapter translation layer to convert MySQL queries (`?`) into PostgreSQL bindings (`$1`) dynamically.

📂 Database Schema Models

Entities, Relational Mapping & Normalization

The data structures are fully normalized to prevent redundancy and keep transactional updates clean.

Core Data Tables

  • products: Name, SKU (Unique), Category, Store, Stock Levels, and Reorder thresholds.
  • alerts: Log entries linking low-stock trigger details to specific product IDs.
  • inventory_history: Tracks old vs new stock quantities on any update.

Relations & Operations

  • suppliers: Registry of third-party grocery distributors.
  • supplier_orders: Logs purchase orders placed for products.
  • sales: Holds past transaction counts and totals for reporting dashboard analytics.

🌐 Frontend Hosting on Vercel

Vite Optimizations & Client Router Configuration

The single-page React app (SPA) is built and optimized into a static bundle, served globally via Vercel's Edge Network.

// vercel.json Rewrite Rules
{
  "version": 2,
  "framework": "vite",
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}

Routing Rewrite: Maps all dynamic routing queries directly back to `index.html` to support internal React Router DOM navigation path lookups.

⚙️ Backend Hosting on Railway

Dockerized Containers & Secrets Injection

The backend app runs on specialized Nixpacks runners, which detect the environment and run a Node.js process wrapper.

nixpacks & railway.json

Instructs Railway to build dependencies cleanly and launch using the start command:

"deploy": {
  "startCommand": "node server.js"
}

Environment Injection

  • `DATABASE_URL`: Cloud PostgreSQL address.
  • `JWT_SECRET`: Security cryptographic hash.
  • `FRONTEND_URL`: CORS configurations origin url.

🚧 Challenges & Resolutions

Deployment Hurdles & Solutions

CORS Blockers

Challenge: Vercel frontend requests to Railway backend were blocked due to different domains.

Solution: Configured dynamic CORS headers inside `server.js` referencing environment configuration variables.

Railway Ephemeral Storage

Challenge: SQLite database file was deleted on container restarts/rebuilds on Railway.

Solution: Migrated from SQLite config to dedicated managed PostgreSQL instances on Railway, securing connection tokens.
